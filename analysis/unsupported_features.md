# Unsupported HDF5 Features

This crate intentionally implements a read-focused subset of HDF5. The table
below tracks known libhdf5 divergences, the supported subset around each
divergence, and the regression tests that currently pin the behavior.

When a row says "coverage gap", the divergence is documented here but still has
an open TODO entry for a dedicated fixture or exact error assertion.

## Dataset Storage And Virtual Datasets

| Area | Current behavior | Regression coverage |
| --- | --- | --- |
| VDS regular hyperslab mappings | Supported for serialized all-selections and regular hyperslabs, including relative source files and same-file source metadata. | `test_reference_virtual_dataset_regular_hyperslabs_read`, `test_virtual_dataset_all_selection_read`, `test_virtual_dataset_same_file_source_read`, `test_virtual_dataset_mixed_all_and_regular_selection_read`, `test_virtual_dataset_fill_value_for_unmapped_regions`, `test_virtual_layout_parses_as_metadata_only` |
| VDS point selections | Supported end-to-end through a real on-disk fixture. Because current `h5py` / libhdf5 creation APIs still reject point mappings in `set_virtual()`, the fixture is generated by patching a valid VDS global-heap mapping object to the exact point-selection wire format after creation. | `test_virtual_dataset_point_selection_read`, `virtual_point_selection_decodes_and_materializes`, `virtual_output_extent_uses_point_and_irregular_destination_bounds` |
| VDS irregular hyperslabs | Supported for serialized irregular hyperslab block-list selections through the VDS mapping read path. | `virtual_irregular_hyperslab_selection_decodes_and_materializes`, `virtual_output_extent_uses_point_and_irregular_destination_bounds`, `test_virtual_dataset_irregular_hyperslab_read` |
| VDS access properties | `HDF5_VDS_PREFIX` environment-variable prefix substitution is honored for source-file lookup, including `${ORIGIN}` expansion. Public `DatasetAccess` exposes `H5Pset_virtual_view` / `H5Pset_virtual_prefix` equivalents for per-call view and prefix behavior without mutating process environment. `VdsMissingSourcePolicy::Fill` allows fixed-shape VDS reads to materialize fill values when source files are absent; the default policy preserves the current I/O error surface. | VDS datatype conversion | VDS source-to-destination materialization uses the crate's shared numeric conversion path for fixed-point/enum/bitfield and floating-point source/destination pairs, including differing element sizes. | v4 chunk indexes | Single-chunk, unfiltered implicit, fixed-array, paged fixed-array, extensible-array, extensible-array spillover, v2-B-tree, and sparse fill-value fallback are supported. | `test_v4_fixed_array_chunks_read`, `test_v4_paged_fixed_array_chunks_read`, `test_v4_filtered_fixed_array_chunks_read`, `test_v4_extensible_array_chunks_read`, `test_v4_extensible_array_spillover_chunks_read`, `test_v4_btree2_chunks_read`, `test_v4_btree2_internal_chunks_read`, `test_sparse_chunked_fill_value_read` |
| Filtered implicit chunk indexes | Explicitly unsupported. HDF5 does not normally choose implicit chunk indexes for filtered datasets, but malformed or hand-authored files can encode them. | `test_filtered_implicit_chunk_index_fixture_is_rejected` patches a real filtered fixed-array fixture into the invalid implicit-index case and asserts the exact unsupported error surface. |
| Mutable chunk writes | `MutableFile::write_chunk` appends/replaces/rebuilds v1 B-tree chunks, replaces existing v4 fixed-array and extensible-array chunks, appends through the currently supported v4 extensible-array direct/data/super-block paths, updates simple v4 v2-B-tree chunk indexes, and rewrites deflate/shuffle/Fletcher32-filtered chunks. Remaining gaps are fixed-array append/growth semantics, deeper v1/v2 B-tree growth beyond the supported rebuilt shapes, and the broader modern-index mutation/growth matrix. | `test_write_chunk_replaces_existing_chunk`, `test_resize_then_write_appended_extensible_array_chunk`, `test_resize_then_write_appended_chunk`, `test_resize_chunked_dataset`, `test_write_chunk_replaces_existing_v4_fixed_array_chunk`, `test_write_chunk_replaces_existing_v4_extensible_array_chunk`, `test_resize_then_write_appended_v4_extensible_array_chunk`, `test_resize_then_write_v4_extensible_array_into_super_block`, `test_resize_then_write_appended_v4_btree2_chunk`, `test_write_chunk_replaces_filtered_chunk`, `test_write_chunk_replaces_fletcher32_chunk`, `test_resize_non_chunked_fails`, `test_resize_wrong_ndims_fails` |
| Undefined storage addresses, late allocation, and external raw data storage | Undefined contiguous storage addresses read as fill values. Late-allocation, fill-time-never contiguous datasets without storage read as HDF5 zero-fill data. External raw data storage is supported for v1 external file-list messages with one or more relative or absolute local paths. External links are decoded separately from external raw data storage. | `test_contiguous_dataset_with_undefined_storage_address_reads_fill_value`, `test_late_allocation_fill_time_never_reads_zeroes`, `test_external_raw_data_storage_reads_relative_file`, `test_external_raw_data_storage_reads_multiple_files` |

| ScaleOffset | Supported for datatype-aware integer and decimal floating-point set-local parameters used by current fixtures, including signed, unsigned, zero-minbits, big-endian, and f32/f64 decimal-scale vectors. | `test_scaleoffset_filter_i32_read`, `test_scaleoffset_filter_big_endian_i32_read`, `test_scaleoffset_filter_f32_read`, `test_scaleoffset_integer_parity_vectors`, `test_scaleoffset_float_parity_vectors`; coverage gap for additional malformed parameter fixtures. |

## Datatypes And Conversion

| Area | Current behavior | Regression coverage |
| --- | --- | --- |
| Primitive conversion | High-level dataset and attribute reads use the shared numeric conversion path for fixed-point, enum/bitfield base bytes, floating-point, integer-float conversion, endian swapping, narrowing, widening, overflow wrapping, NaN, and infinity behavior covered by current fixtures. Public datatype inspection exposes numeric bit offset/precision and floating-point field/bias/normalization/padding metadata. Full libhdf5 conversion-path selection parity remains incomplete. | Existing primitive reference tests in `tests/t3_datatypes.rs`, including `t3a_integer_datatype_precision_and_offset` and `t3a_float32_field_metadata`, plus `test_attr_read_typed` and `test_attr_read_array_typed`; coverage gap for the full libhdf5 conversion matrix called out in TODO. |
| Compound fields | Metadata preserves member datatypes, byte order, offsets, nested datatypes, and member names. `read_field::<T>()` supports fixed-size fields with matching Rust size. `read_field_raw()` returns raw member bytes for non-primitive members. | `test_compound_dtype_info`, `test_compound_read_field_f64`, `test_compound_read_field_i32`, `test_compound_read_field_raw`, `test_compound_read_field_wrong_size`, `test_compound_fields_api`, `test_compound_field_preserves_member_byte_order`, `test_compound_fields_reject_truncated_member_metadata` |
| Recursive compound values | Nested compound, array, variable-length, and reference members can be decoded through `read_field_values()`. | Variable-length strings | Vlen string datasets and attributes backed by global heap references are readable through high-level string APIs, including empty strings, null descriptors, UTF-8 payloads, duplicate heap objects, deleted/free global-heap entries, and larger heap-edge payloads. | `test_read_vlen_string_attr`, `test_read_vlen_string_dataset`, `test_variable_length_attribute_payload_raw_read`, `t9a_global_heap_vlen_strings`, `t9a_global_heap_vlen_attr`, `t3d_vlen_string_empty_null_utf8_and_heap_edges`, `t9a_global_heap_deleted_objects_duplicate_ids_and_padding`, `t9a_global_heap_read_object_skips_deleted_and_padding`, `test_vlen_base_distinguishes_sequence_and_string_metadata`, `test_vlen_base_rejects_truncated_or_ambiguous_metadata` |

| --- | --- | --- |
| External links | Link messages decode external filename/object path, writer APIs can create external links, and read-side traversal opens target files for same-directory, relative, and absolute target paths. | Soft-link traversal | Basic link metadata is decoded, public link iteration exposes hard/soft/external link records, writer APIs can create hard/soft/external links, tracked creation-order link iteration is available when the file records link creation order, and read-side traversal has bounded cycle detection. | `test_write_and_read_hard_links`, `test_write_and_read_soft_link`, `test_write_external_link`, `test_read_dense_links`, `test_dense_group_creation_order_indexing_enabled_and_disabled`, `test_soft_link_resolution_and_cycle_limit`; coverage gap for deeper libhdf5 path normalization edge cases. |

## Writer Surface

| Area | Current behavior | Regression coverage |
| --- | --- | --- |
| Dataset creation | Writer support is intentionally narrow: primitive dataset creation, compact fixed-string, compound, enum, opaque, array, and contiguous vlen UTF-8 string datasets, scalar/array/fixed-string/fixed-string-array compact and dense attributes on root groups, groups, and datasets, explicit fill-value messages, writer-created modern chunk indexes, and selected mutable chunk operations are supported. | Existing writer tests in `tests/write_test.rs`, `tests/write_api_test.rs`, `tests/attr_write_test.rs`, `tests/resize_test.rs`, and `tests/link_test.rs`; coverage gap for the remaining writer parity and h5dump/h5py round-trip items in TODO. |
| Writer chunk indexes | New chunked datasets are written with v4 single-chunk, fixed-array, extensible-array, or v2-B-tree chunk indexes depending on shape, max shape, and chunk count. `MutableFile::write_chunk` can append/replace/rebuild v1 B-tree chunks, replace existing v4 fixed-array and extensible-array chunks, append/update selected v4 extensible-array chunks, and update simple v4 v2-B-tree chunk indexes. Remaining gaps are the full modern chunk-index mutation matrix and deeper B-tree growth cases. | `test_write_chunk_replaces_existing_chunk`, `test_resize_then_write_appended_extensible_array_chunk`, `test_resize_then_write_appended_chunk`, `test_write_chunk_replaces_existing_v4_fixed_array_chunk`, `test_write_chunk_replaces_existing_v4_extensible_array_chunk`, `test_resize_then_write_appended_v4_extensible_array_chunk`, `test_resize_then_write_v4_extensible_array_into_super_block`, `test_resize_then_write_appended_v4_btree2_chunk`. |
| Free-space managers | Write paths append new metadata/raw data and do not implement libhdf5 free-space manager reuse. File growth and address reuse are therefore not a compatibility guarantee. | Documented behavior; no free-space reuse tests are expected until a free-space manager writer is added. |

## Out-Of-Scope Subsystems

These families still appear in `ccc-rs missing`, but they are not accidental
translation holes in the current crate scope.

| Area | Current behavior | Regression coverage |
| --- | --- | --- |
| VOL / async / plugin / connector stack | Not implemented. The crate exposes a native Rust API rather than mirroring libhdf5's VOL, async-event-set, and plugin infrastructure. | No tests planned; see `analysis/ccc_missing_roadmap.md`. |
| MPI / parallel HDF5 | Not implemented. We will not use MPI-backed HDF5 in this crate. If CPU parallelism is introduced later, it should be a Rust-side Rayon optimization after behavior is pinned. | No tests planned; intentionally out of scope. |
| Alternative VFDs and network/cloud drivers | Not implemented beyond the ordinary local-file access used by the crate today. HDFS, ROS3, Direct, Core, and stdio driver parity are outside the current surface. | No tests planned; intentionally out of scope. |
| Full dataspace selector iterator parity | Dataset/dataspace decode, common slice reads, `none`, explicit points, regular block hyperslabs, stepped slices, bounded point materialization, set-combine helpers, selection count/bounds/regularity/contiguity/linear-span helpers, VDS decode, and selected materialization paths are supported. Full libhdf5 iterator/projection parity remains incomplete. | `selection_test`, `t7_selections`, and VDS tests; coverage gap for deeper selector iterator/projection parity remains tracked in TODO. |

## Tracehash Parity Boundary

The old tracehash corpus workflow has been retired because its patched C
harness and Rust corpus walker no longer matched the current crate surface.
The remaining `tracehash` Cargo feature is a local-debug probe hook only, not a
supported compatibility claim. See `analysis/tracehash_divergence_report.md`.
